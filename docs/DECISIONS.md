# Architectural Decision Records (ADR)

> Документирование ключевых архитектурных решений проекта.

## ADR-001: Использование asset-pack архитектуры для зданий

**Статус:** Принято  
**Дата:** 2026-01-XX  
**Авторы:** Команда разработки

### Контекст

Проект требует поддержки разных культур (Россия, Европа, Османы) и стадий строительства зданий без хардкода в игровом коде.

### Решение

Использовать metadata-driven архитектуру:
- Метаданные хранятся в `src/assets/buildings/<country>/<building>/meta.json`
- Рендеринг осуществляется через `BuildingVisualSet` с container root
- Регистрация ассетов в `buildingAssetRegistry.ts`

### Последствия

**Плюсы:**
- Легко добавлять новые культуры и здания
- Разделение кода и контента
- Поддержка разных стадий строительства
- Готовность к SVG/PNG/WebP текстурам

**Минусы:**
- Требует создания meta.json для каждого здания
- Дополнительный слой абстракции

### Пример

```json
{
  "kind": "town_center",
  "country": "russia",
  "stages": {
    "construction_0": { "layers": ["shadow", "base", "scaffolding"] },
    "complete": { "layers": ["shadow", "base", "roof", "details"] }
  }
}
```

## ADR-002: BuildingVisualSet с container root

**Статус:** Принято  
**Дата:** 2026-01-XX

### Контекст

Раньше части здания (основание, крыша, тени) позиционировались независимо, что приводило к визуальным артефактам при синхронизации.

### Решение

Использовать Phaser Container как корневой элемент:
```typescript
const root = scene.add.container(x, y);
root.add(basePart);
root.add(roofPart);
root.add(shadowPart);
```

Все части становятся детьми контейнера, позиционирование относительно центра здания.

### Последствия

**Плюсы:**
- Все части движутся вместе
- Простая синхронизация через `root.setPosition()`
- Легкое управление прозрачностью всего здания

**Минусы:**
- Требует переписывания старого кода

### Паттерн использования

```typescript
// ✅ ПРАВИЛЬНО
building.visuals.root.setAlpha(0.5);
building.visuals = syncBuildingVisuals(...);

// ❌ НЕПРАВИЛЬНО
building.visuals.forEach(part => part.setPosition(x, y));
```

## ADR-003: Отказ от визуальных индикаторов строительства

**Статус:** Принято  
**Дата:** 2026-01-XX

### Контекст

Ранее использовался эмодзи молотка над рабочими, что создавало визуальный шум и требовало дополнительного рендеринга.

### Решение

Убрать молоток. Статус строительства отображается:
- Через поведение рабочего (идёт к стройке, работает)
- Через UI (панель портретов показывает "Возводит здание")
- Через цвет рамки здания (оранжевый — в процессе)

### Последствия

**Плюсы:**
- Чище визуальный ряд
- Меньше кода рендеринга
- Естественный геймплей

**Минусы:**
- Игроку нужно научиться читать индикаторы UI

## ADR-004: Монолитная BattleScene.ts

**Статус:** Временное решение  
**Дата:** 2026-01-XX

### Контекст

BattleScene.ts стал слишком большим и сжатым для оптимизации скорости разработки.

### Решение

Оставить как есть до стабильности ядра. Не рефакторить без явного запроса.

### Последствия

**Плюсы:**
- Быстрая итерация разработки
- Все в одном месте

**Минусы:**
- Сложно поддерживать
- Высокий риск при изменениях

### План на будущее

Разделить на модули:
- `selectionSystem.ts`
- `buildingPlacement.ts`
- `constructionSystem.ts`
- `combatSystem.ts`
- `hudSystem.ts`
- `minimapSystem.ts`

## ADR-005: Изометрический рендеринг на Canvas

**Статус:** Принято  
**Дата:** 2026-01-XX

### Контекст

Требуется 2.5D вид для исторической RTS.

### Решение

Использовать изометрическую проекцию на Canvas через Phaser:
- `IsoUtils` для преобразования координат
- Depth sorting по Y-координате
- Layered rendering (ландшафт → здания → юниты → UI)

### Альтернативы

- WebGL с кастомными шейдерами (сложнее)
- 2D top-down (проще, но менее атмосферно)
- 3D модель (слишком сложно для прототипа)

## ADR-006: TypeScript + Phaser 3 + Vite

**Статус:** Принято  
**Дата:** 2026-01-XX

### Контекст

Выбор стека для браузерной RTS.

### Решение

- **TypeScript** — типобезопасность, рефакторинг
- **Phaser 3** — готовый игровой фреймворк
- **Vite** — быстрый dev server и билд

### Альтернативы

- **Three.js** — для 3D, но сложнее для 2.5D
- **Babylon.js** — мощный, но избыточен
- **Кастомный движок** — слишком долго

## ADR-007: Система ресурсов

**Статус:** Принято  
**Дата:** 2026-01-XX

### Контекст

Нужна экономика для рекрутинга и строительства.

### Решение

4 ресурса + население:
- 🍞 **Food** — пища (поддержание армии)
- 🪵 **Wood** — дерево (здания)
- ⛏ **Iron** — железо (оружие, артиллерия)
- 🪙 **Gold** — золото (элитные юниты)
- 👥 **Population** — лимит юнитов

### Генерация

Ресурсы генерируются с регионов на стратегической карте каждый ход.

## ADR-008: Формации войск

**Статус:** В разработке  
**Дата:** 2026-01-XX

### Контекст

Исторические RTS требуют построения отрядов в формации.

### Решение

Поддерживаемые формации:
- **Line** — линия (максимальный залп)
- **Column** — колонна (движение)
- **Square** — квадрат (защита от кавалерии)
- **Wedge** — клин (атака)

### Механика

- Скорость движения снижается в формации
- Бонусы к бою в зависимости от формации
- Поворот формации на 90°
